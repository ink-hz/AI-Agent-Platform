# AI Agent 集群查看平台产品设计

- 日期：2026-07-21
- 状态：已完成产品讨论，待用户审阅书面规格
- 产品定位：单用户、本机、只读的 AI Agent 团队驾驶舱

## 1. 背景

当前 Platform 已经能够从 MetaBot 运行契约发现 9 个 Agent，并通过健康接口展示在线状态、运行时长和技术探测数据。现有页面仍然站在服务器监控视角：端口、延迟、进程名占据主要位置，无法回答 Agent 被如何使用、做了什么、答案如何生成以及数据飞轮积累了什么。

Orbbec-Agent-Team 已经具备一套真实运行的数据飞轮：9 个 Bot 共用本机 PostgreSQL，通过 `bot_id + business_domain` 隔离；系统记录会话、完整消息、运行、工具调用、证据、反馈，并生成改进候选和回归案例。Platform 不需要重做采集链路，只需要把现有数据组织成可查看的产品。

## 2. 产品定义

Platform 从“MetaBot 技术监控页”升级为“AI 团队驾驶舱”。它首先回答：

1. 已经开发了多少个 Agent；
2. 每个 Agent 当前状态如何、持续运行了多久；
3. 每个 Agent 已经完成了多少次真实对话，最近是否仍在被使用；
4. 某个 Agent 最近承担了哪些任务、有哪些会话、给出了什么答案；
5. 某条答案使用了哪些知识、工具和执行步骤；
6. 数据飞轮为该 Agent 积累了哪些反馈、改进候选和回归案例。

第一版仅展示和查看，不控制 Agent，不修改任何运行或飞轮数据。

## 3. 用户与部署边界

- 当前唯一用户是本机所有者，不实现登录、角色管理或 RBAC 页面。
- Platform 继续只监听 `127.0.0.1`，不作为局域网或公网服务开放。
- 数据库连接继续使用现有只读角色；不因为 UI 无权限系统而放宽数据库权限。
- Platform 不向 MetaBot 下发任务、重启进程、修改配置或调用控制接口。
- 页面展示的使用量必须来自真实数据。Canary、测试探针和其他 `is_synthetic` 数据不得进入正式统计。
- 线框稿中的数字属于明确标注的界面示意数据，不进入生产页面。

## 4. 非目标

本期不实现：

- Agent 聊天入口；
- Agent 启停、重启、任务下发或自动恢复；
- 提示词、知识库、工具或工作流编辑；
- 改进候选审批、自动应用或版本发布；
- 告警通知；
- 多用户、权限管控或远程访问；
- 长期历史监控系统、指标告警平台或新的数据仓库；
- 伪造或放大真实使用量。

## 5. 产品原则

### 5.1 使用价值优先

首页的视觉主角是 Agent 数量、真实使用量和最近活动。状态和运行时长用于解释 Agent 是否可用。端口、探测延迟和进程名只进入二级运行诊断，不占据首页卡片。

### 5.2 一张卡代表一个 Agent

每个 Agent 具有稳定的名称、图标、业务领域和一句职责介绍。卡片不是服务器资源块，而是一个数字员工档案摘要。

### 5.3 渐进式查看

查看路径固定为：

`集群总览 → Agent 详情 → 任务或会话 → 对话与答案 → 执行链路与知识证据`

用户不需要在互不关联的日志、消息和数据库记录之间自行拼接上下文。

### 5.4 结果与解释同行

查看一条最终答案时，完整对话放在左侧，知识引用、工具调用、证据和执行步骤放在右侧。答案与生成链路不得拆成无法关联的独立页面。

### 5.5 真实与可追溯

每个汇总数字都必须有明确口径，并能逐层下钻到组成它的真实记录。无法获取的数据标记为不可用，不使用 `0` 冒充。

## 6. 信息架构

顶层导航包含：

1. **总览**：AI 团队驾驶舱；
2. **Agents**：全部 Agent 列表与 Agent 详情；
3. **任务与会话**：跨 Agent 查看近期工作记录；
4. **数据飞轮**：跨 Agent 查看反馈、改进候选和回归案例。

Agent 详情内部包含：

1. 概览；
2. 任务；
3. 会话与答案；
4. 执行链路；
5. 知识库；
6. 数据飞轮。

“对话”和“答案”不是两个平级模块。会话是容器，包含用户与 Agent 消息；每个已完成 turn 对应一条 Agent 答案；执行链路通过 `turn_id` 和 `run_id` 解释该答案的生成过程。

## 7. 集群总览

### 7.1 顶部摘要

展示四个核心指标：

- 已开发 Agent 数量；
- 当前运行中的 Agent 数量；
- 累计真实对话次数；
- 最近 7 天真实对话次数及与前 7 天的变化。

摘要下方展示：

- 最近 7 天全部 Agent 的对话趋势；
- 按累计对话次数排序的活跃 Agent；
- 数据最后更新时间。

### 7.2 Agent 卡片

每张卡片展示：

- 名称、图标、业务领域和一句职责介绍；
- 当前产品状态；
- 累计真实对话次数；
- 最近 7 天真实对话次数；
- 当前连续运行时长；
- 最近一次真实活动时间；
- 最近任务或会话的短摘要；
- 进入只读详情的链接。

不展示端口、延迟、进程名或健康 URL。

### 7.3 产品状态口径

状态同时使用运行健康和最近活动推导：

- `活跃`：运行健康，且最近 15 分钟内有非合成消息或 run；
- `在线`：运行健康，但最近 15 分钟没有真实活动；
- `异常`：实例可达但健康响应异常；
- `离线`：连接失败或探测超时；
- `检测中`：尚未完成第一次健康探测；
- `数据过期`：状态快照超过 30 秒未更新。这是数据新鲜度，不替代实例状态。

15 分钟活跃窗口作为 Platform 配置项保存，首版默认值固定为 15 分钟。

### 7.4 对话次数口径

“一次对话”定义为一个真实用户 turn 获得了一条已落库的 assistant 消息：

- 以非合成数据中的唯一 `turn_id` 计数；
- 必须存在 `role = 'assistant'` 的消息，代表答案已经实际生成并记录；
- 用户追问产生新的 `turn_id`，计为新的一次对话；
- 单个飞书会话容器可以包含多次对话；
- 失败且没有生成 assistant 消息的 run 不进入对话次数；
- `is_synthetic = true` 的 canary、探针和测试数据不进入任何使用量统计。

该口径比直接统计 `flywheel_core.conversations` 更能表达 Agent 被实际使用的次数。

## 8. Agent 详情

### 8.1 详情头部

展示 Agent 身份、职责介绍、业务领域、接入渠道、当前状态、累计对话、本周对话、运行时长和最近活动。

Agent 的友好名称、图标、领域、介绍和能力来自 Platform 本地只读展示目录，以 `bot_id` 为键。运行契约仍然是实例发现真相源；未知 Bot 使用通用回退身份，不阻塞展示。

### 8.2 概览

概览组合展示：

- 最近 7 天使用趋势；
- 最近会话与答案摘要；
- 最近任务；
- Agent 介绍与能力；
- 已配置知识库摘要；
- 数据飞轮摘要。

### 8.3 任务

首版的任务记录由 flywheel `runs` 和明确的任务事件归一化而来：

- 有稳定外部任务标识时，以任务标识聚合多个 run；
- 没有任务标识时，一个顶层 run 作为一条任务记录；
- 展示任务标题或输入摘要、状态、开始/完成时间、所属会话、产出答案和错误；
- 不在 Platform 中创建或修改任务。

## 9. 会话、答案与执行链路

### 9.1 会话列表

按 Agent 和时间分页展示：

- 会话标题或首条用户问题摘要；
- 渠道和会话类型；
- 对话轮次数；
- 最近消息时间；
- 答案、知识引用、工具调用和错误数量。

默认过滤真实数据，可按时间和状态检索。查询不一次加载全部消息正文。

### 9.2 会话详情

左侧时间线展示该会话中的完整用户消息和 Agent 消息。选中任一 assistant 消息后：

- 标识对应 `turn_id`；
- 展示完整最终答案；
- 展示回答时间和关联反馈；
- 右侧检查器加载对应 run 和执行链路。

### 9.3 执行链路

右侧检查器依次展示：

1. 用户意图或执行开始；
2. 知识检索事件；
3. 工具调用及结构化结果；
4. 模型执行摘要；
5. 最终答案；
6. 错误、重试和完成状态。

默认读取 `flywheel_trace.runs` 和 `flywheel_trace.events` 的结构化数据。原始 L3 evidence 只在用户明确展开“原始证据”时按需读取，不进入列表、首页或预取缓存。

### 9.4 知识引用

知识查看分为两部分：

- **Agent 已配置的知识**：后端只读扫描运行契约允许的 `knowledge/` 目录，返回逻辑名称、文件类型、更新时间和版本摘要；不向页面暴露真实文件系统路径；
- **回答实际使用的知识**：从 trace 事件、sources 和 evidence 关联到当前 `turn_id`，展示引用片段、来源名称和验证状态。

如果知识目录无法读取，Agent 的状态、会话和答案仍可正常展示；知识区域单独标记不可用。

## 10. 数据飞轮查看

数据飞轮页面和 Agent 详情中的飞轮区域均为只读，展示：

- 标准化反馈信号；
- 改进候选项及严重度、优先级、出现次数和证据链；
- 回归测试草稿；
- 数据处理最近成功时间；
- 原始证据留存状态和数据缺口状态。

数据来自 `flywheel_governance` 的只读视图，包括 `feedback_signal_summary`、`open_improvement_candidates`、`candidate_evidence_chain`、`regression_case_queue` 和 `raw_retention_status`。

页面不提供接受、拒绝、应用、编辑或发布操作。

## 11. 数据架构

### 11.1 数据源

- 运行契约和现有健康轮询器：Agent 清单、状态和运行时长；
- `flywheel_analytics`：自动排除 synthetic 数据的 conversations、messages、runs 和 events；
- `flywheel_governance`：反馈、候选、回归和治理健康；
- `flywheel_evidence`：按需读取的高敏感原始证据；
- Platform 展示目录：Agent 友好身份、介绍和能力；
- Agent knowledge 目录：只读知识清单和版本信息。

### 11.2 Platform 后端模块

后端增加相互独立的只读组件：

1. `FlywheelReadRepository`：通过 `flywheel_analyst` 读取 analytics 和 governance 视图；
2. `EvidenceReadRepository`：仅在配置独立 evidence reader 凭据时按 ID 读取单条证据；
3. `AgentCatalog`：合并运行契约与展示目录；
4. `KnowledgeCatalog`：安全列出 Agent 允许的知识目录；
5. `FleetReadService`：统一统计口径、状态推导和跨源组合；
6. 分页查询 API：向前端提供稳定的产品 Read Model。

前端不直接依赖数据库表名或 flywheel schema。

### 11.3 只读 API 边界

产品 API 至少包括：

- `GET /api/fleet/overview`
- `GET /api/fleet/agents`
- `GET /api/fleet/agents/{bot_id}`
- `GET /api/fleet/agents/{bot_id}/tasks`
- `GET /api/fleet/agents/{bot_id}/conversations`
- `GET /api/fleet/conversations/{conversation_id}`
- `GET /api/fleet/turns/{turn_id}/trace`
- `GET /api/fleet/agents/{bot_id}/knowledge`
- `GET /api/fleet/agents/{bot_id}/flywheel`
- `GET /api/fleet/flywheel`

所有 Agent 级查询显式包含 `bot_id` 过滤。列表使用游标或稳定时间分页，限制正文和 JSONB 大字段的默认返回量。

### 11.4 缓存和刷新

- 运行状态继续每 10 秒刷新；
- 首页使用量汇总缓存 60 秒；
- 详情摘要缓存 60 秒；
- 会话、任务、链路、知识和飞轮列表按需查询；
- 会话正文和原始证据不进入共享预取缓存；
- 页面始终显示最后成功更新时间。

Platform 不复制一套新的长期业务数据库。缓存只存在于进程内，重启后重新读取真实数据源。

## 12. 降级与错误处理

不同数据源必须独立降级：

- 健康探测失败：保留最后状态并标记过期；
- PostgreSQL 不可用：首页仍展示 Agent 清单和运行状态，使用量显示“暂不可用”；
- analytics 查询失败：不以 `0` 替代真实统计；
- governance 查询失败：只隐藏或标记飞轮区域，不影响会话；
- knowledge 目录不可读：只影响知识目录区域；
- 单条 trace 或 evidence 不完整：展示已有链路并明确标注缺失步骤；
- 未知 Bot：使用运行契约名称和通用介绍继续展示；
- 请求失败后，前端保留最后一次成功数据并显示时间，不清空整个页面。

错误信息必须简短、安全，不返回数据库连接、凭据、真实主机路径或堆栈。

## 13. 安全与隐私

- Platform 仅绑定本机回环地址；
- 数据库 URL 从 Keychain 或进程安全配置读取，不进入仓库、LaunchAgent plist 或页面；
- 默认读取 `flywheel_analyst` 可见的结构化数据；
- L3 原始证据使用独立只读凭据并按需访问；
- 页面不显示数据库主键以外的秘密、文件系统绝对路径或内部凭据；
- thinking / chain-of-thought 不存储也不展示；执行链路只展示结构化事件摘要；
- 虽然当前不做 UI 权限系统，代码边界仍保留 `bot_id` 隔离，避免不同 Agent 数据串台。

## 14. 前端状态

每个主要页面都必须覆盖：

- 初次加载；
- 有数据；
- 空数据；
- 部分数据源不可用；
- 数据过期；
- 单条记录缺少关联链路；
- 小屏布局。

颜色不是唯一状态信号。状态、错误和过期都必须同时显示文字。

## 15. 测试策略

### 15.1 数据口径测试

- 对话次数按唯一真实 `turn_id + assistant message` 计算；
- synthetic 数据不进入累计、趋势和排名；
- 失败且无答案的 run 不计入对话次数；
- 最近 7 天与前 7 天边界正确；
- 每个 Agent 查询始终应用 `bot_id`；
- 活跃、在线、异常、离线、检测中和过期推导正确。

### 15.2 Repository 和 API 测试

- 使用隔离 PostgreSQL fixture 验证 analytics 和 governance 查询；
- 分页稳定且无重复或遗漏；
- 数据源部分失败时 API 返回可解释的 partial 状态；
- knowledge 路径限制阻止越过允许目录；
- API 不泄漏凭据、工作目录和原始异常；
- evidence 未配置时结构化链路仍可使用。

### 15.3 前端测试

- 首页摘要、趋势、排名和 Agent 卡片使用统一口径；
- Agent 详情能进入正确的会话；
- assistant 消息与对应 `turn_id` 链路正确关联；
- 任务、会话、知识和飞轮的空状态与失败状态正确；
- 页面无控制、编辑、重启或发布操作；
- 响应式布局和文字对比度满足现有质量门槛。

### 15.4 本机验收

- Platform LaunchAgent 正常运行；
- 9 个运行契约 Agent 全部出现在总览；
- 页面状态与健康探测快照一致；
- 使用量能下钻到组成统计的真实 assistant 消息；
- 任取一个真实会话，可查看完整消息、答案、run、事件和知识证据；
- 任取一个 Agent，可查看反馈、改进候选和回归案例；
- 断开 PostgreSQL 时，Agent 状态监控仍可使用；
- Platform 构建、前后端测试和依赖审计通过；
- 部署过程不重启或修改 MetaBot。

## 16. 交付顺序

为保证每一阶段都能独立使用，按以下顺序交付：

1. **只读数据底座与团队总览**：连接 analytics/governance，建立统一统计口径，交付首页和 Agent 卡片；
2. **Agent 详情与会话答案**：交付 Agent 档案、使用趋势、任务、会话列表和完整消息；
3. **执行链路、知识与飞轮**：交付 turn/run/event 关联、知识目录、证据查看和治理页面；
4. **生产验收与收口**：验证真实数据、降级行为、性能、LaunchAgent 和 MetaBot 无干扰。

每个阶段都保持只读，并在真实数据不足或关联缺失时展示事实，不伪造内容。

## 17. 成功标准

用户打开 Platform 后能够在一个页面内看出：已经开发了多少 Agent、哪些正在运行、运行了多久、真实完成了多少次对话以及哪些 Agent 使用最活跃。

用户进入任一 Agent 后，可以沿着 Agent → 任务/会话 → 对话/答案 → 执行链路/知识证据查看真实工作记录，并能看到该 Agent 的反馈、改进候选和回归案例。整个过程不需要进入数据库、日志目录或 MetaBot 工作目录，也不会改变任何 Agent 或飞轮数据。
